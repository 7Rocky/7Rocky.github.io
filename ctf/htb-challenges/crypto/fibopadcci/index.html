class RequestHandler(socketserver.BaseRequestHandler):
    handle = handle

    def read(self, until='\n'):
        out = ''
        while not out.endswith(until):
            out += self.request.recv(1).decode()
        return out[:-len(until)]

    def query(self, string=''):
        self.write(string, newline=False)
        return self.read()

    def write(self, string, newline=True):
        self.request.sendall(str.encode(string))
        if newline:
            self.request.sendall(b'\n')

    def close(self):
        self.request.close()

Un Padding Oracle Attack funciona de la siguiente manera:

Podemos ajustar el segundo bloque de texto cifrado, que afecta directamente al √∫ltimo bloque de texto claro. La idea es iterar el √∫ltimo byte desde 0x00 hasta 0xff Hasta que encontremos uno que resulte en 0x01 (el primer byte de la secuencia de relleno). En este punto, el servidor mostrar√° un mensaje exitoso.

Sin embargo, este no es un Padding Oracle Attack t√≠pico contra AES CBC. En tal caso, encontrar√≠amos un byte “m√°gico” ($B_i$) que no da error de relleno y se cumplen estas condiciones:

$$ B_i \oplus \mathrm{pt}_i = \mathrm{pad}_i \iff B_i \oplus \mathrm{AES.dec}(\mathrm{ct}_i) = \mathrm{pad}_i $$

Entonces, el byte de texto claro $\mathrm{pt}_i$ se calcula como:

$$ \mathrm{pt}_i = \mathrm{AES.dec}(\mathrm{ct}_i) = \mathrm{pad}_i \oplus B_i $$

Esta vez, el texto cifrado no se env√≠a directamente al descifrador, hay una operaci√≥n XOR con el bloque anterior de texto claro. Sin embargo, es necesario comprender los conceptos detr√°s del Padding Oracle Attack cl√°sico para comprender el proceso para resolver este reto. Haremos lo siguiente:

Con esta configuraci√≥n, si usamos el valor proporcionado de b, el mensaje se descifrar√≠a correctamente. Entonces, si obligamos al √∫ltimo byte a ser 0x01 (en este caso concreto), el servidor no dar√° error y podremos hacer los c√°lculos anteriores para encontrar el byte de texto sin formato.
