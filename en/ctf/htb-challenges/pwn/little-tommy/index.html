Little Tommy


We are given a 32-bit binary called little_tommy:

Arch:     i386-32-little
RELRO:    Partial RELRO
Stack:    No canary found
NX:       NX enabled
PIE:      No PIE (0x8048000)

We do not have the C source code, but we can obtain a decompiled source code using a reverse engineering tool like Ghidra. This is the main function:

Basically, the program provides a menu with these options:

$ ./little_tommy

#################### Welcome to Little Tommy's Handy yet Elegant and Advanced Program ####################  

1. Create account
2. Display account
3. Delete account
4. Add memo
5. Print flag

Please enter an operation number:

The data is stored on the heap using malloc. Moreover, we can release an object using free.

The program is vulnerable to Use After Free because we can add a memo, and this string will be stored on the heap as well using strdup (which calls malloc behind the hoods). So, this is a heap exploitation challenge. As a curiosity, the acronym for “Handy yet Elegant and Advanced Program” is HEAP.

The idea is to create an account, delete it and add a memo so that we can overwrite the previous chunk with the data necessary to retrieve the flag:

case '5':
  if ((main_account == (char *) 0x0) || (*(int *) (main_account + 0x40) != 0x6b637566)) {  
    puts("\nNope.");
  } else {
    system("/bin/cat flag");
  }

To exploit the Use After Free vulnerability, we need that the chunk allocated by strdup has 0x51 as chunk metadata. If this happens, malloc will reuse the previously released chunk to store this new data because it has the same size (instead of allocating a new chunk).

In the first visualization of the heap, we can see that the first and last names have 32 bytes reseerved each. Hence, the memo we need to enter must have a length between 60 and 75 (approximately). Let’s test it:
